### 5. Sleep/Wake Recovery (the fix for "why is Koda not working again")
- **Root cause**: Windows sleep/wake kills audio devices and keyboard hooks silently. The old watchdog only checked if the stream/subprocess was "alive" — but Windows reports them as alive even when they're dead.
- **Fix**: Watchdog now tracks `time.monotonic()` between checks. If a 15-second sleep took >30 seconds wall time, it means the system was asleep. Triggers `_full_recovery()`:
  1. Waits 2s for hardware to stabilize
  2. Tears down and recreates audio stream (`sd.InputStream`)
  3. Kills and restarts hotkey service subprocess
  4. Updates tray to green "Ready"
- Koda recovers within ~17 seconds of wake/unlock

## Decisions Made

1. **Sleep/wake detection via time gaps, not Windows messages** — tracking `time.monotonic()` is simpler and more reliable than hooking `WM_POWERBROADCAST` from a background process. If a 15s sleep takes >30s, the system was asleep.
2. **Full recovery on wake, not partial** — both audio stream AND hotkey service get restarted, because both die after sleep. Partial recovery leaves things broken.
3. **No install until dev is done** — user explicitly doesn't want to keep reinstalling. Running from source with `pythonw voice.py`. Startup shortcut was attempted but user hasn't approved it yet.
4. **Dropdown pickers, not free-text** — users can only pick from safe F-key options. Prevents the conflict issues from session 7 (ctrl+shift+P, ctrl+backtick, etc.).
